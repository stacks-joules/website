# Gatsby to Astro Migration Plan

## Stacks+Joules Website
**Repo:** github.com/stacks-joules/website
**Date:** 2026-03-30
**Estimated effort:** 12-16 hours across 7 phases

---

## Why Migrate?

- Gatsby is effectively abandoned (no updates since 2024)
- 46 npm vulnerabilities, almost all from Gatsby's dependency tree
- `gatsby develop` broken on Node 22+ without workarounds
- The site is a static marketing site - Astro is purpose-built for this
- Astro ships near-zero JS to the browser (only hydrates interactive bits)

## Why This Migration is Clean

- **Zero GraphQL** - all content is hardcoded in components
- **No gatsby-plugin-image** - just plain `<img>` tags
- **Only 2 files** import from `gatsby` (NavLink.tsx and Card.tsx use `Link`)
- **No SCSS** despite sass plugin being installed
- **Serverless function** is framework-agnostic (stays as-is)

## Phase 1: Layout & Structural Components
**~3-4 hours | Medium risk**

- [ ] Create `src/layouts/Layout.astro`:
  - Import `global.css` and `fonts.css` in `<head>`
  - Add font preload `<link>` (currently in `gatsby-ssr.js`)
  - Add SEO meta tags (react-helmet is installed but never used)
  - Render Header as `<Header client:load />`
  - Render Navigation as `<Navigation client:load />`
  - Render `<slot />` for page content
  - Render ContactSection as `<ContactSection client:visible />`
  - Render Footer as inline static HTML
- [ ] Create `src/data/site.ts` with site metadata constants from `gatsby-config.ts`
- [ ] Keep ThemeProvider bundled inside Header island (React context can't span islands)
- [ ] Replace `import { Link } from 'gatsby'` in:
  - `src/components/common/NavLink.tsx` → plain `<a>` tag
  - `src/components/blocks/card-section/Card.tsx` → plain `<a>` tag
- [ ] Implement active link styling using `window.location.pathname` instead of Gatsby's `activeClassName`

### Island Architecture

**Hydrate immediately (`client:load`):**
- Header (WebGL shader background, must render above the fold)
- Navigation (mobile menu toggle, scroll listener)

**Hydrate when visible (`client:visible`):**
- ContactSection (form with fetch to serverless function)
- NewsletterSignup (form submission)
- AccordionPanel (expand/collapse)

**Pure static (no hydration):**
- Container, SectionHeader, Footer, Buffer, Wallpaper
- ImagePanel, TextPanel, MarkupPanel, Block
- CardContainer, Infographic

**Gotcha:** ThemeProvider uses React context which cannot span Astro islands. Currently only Header consumes it - keep ThemeProvider inside the Header island. If cross-island state is needed later, switch to `nanostores`.

## Risk Summary

| Risk | Impact | Mitigation |
|------|--------|------------|
| React context can't span islands | ThemeProvider breaks if used across components | Keep it inside Header island only |
| Netlify Forms detection | Newsletter form may stop working | Convert to serverless function |
| Build output dir (`public/` → `dist/`) | Deploy serves 404s | Add `netlify.toml` with correct publish dir |
| WebGL shader (NeuroNoiseBackground) | Must hydrate immediately or hero is blank | Use `client:load` on Header island |
| Font paths | Could break if not mapped correctly | `static/fonts/` → `public/fonts/` preserves same `/fonts/...` paths |

---

## Files That Need No Changes

- `netlify/functions/submitForm.js` — framework-agnostic
- All CSS Module files (17 files) — Astro supports CSS Modules natively
- `src/styles/global.css` and `src/styles/fonts.css` — direct copy
- All interactive React components — become Astro islands with hydration directives
